# 题目词元锚定的潜状态通信方案

日期：2026-07-09

工程状态（2026-07-12）：`scripts/run_mca_question_token_anchored_delta.py` 和对应测试已经存在；旧 `A800_2` 上的 50 题等待器未产生结果，且旧机公钥已失效。后续运行默认迁移到逻辑远端 `X_LANCE_HPC`，但必须先确认物理超算并完成 SSH 首登、GPU 显存和运行参数兼容性检查。

## 问题

我们当前要回答的问题是：

```text
在多智能体推理中，能否不新增可读通信文本，也不要求智能体改变原始推理提示词，
而是把一个智能体推理过程中产生的潜状态变化，按语义位置注入到另一个智能体读取同一题目时的对应位置？
```

这个问题来自此前两个失败或不充分的方向：

1. 只强调“答案生成前通信”不够新。已有工作已经研究了推理中途的文本通信和潜状态通信。
2. 只强调“非文本通信”也不够新。已有工作已经传过嵌入、隐状态差分、键值缓存和压缩状态。

因此，本方案不再把创新点放在“通信介质是非文本”或“通信时机在答案前”，而是放在一个更具体的问题上：

```text
非文本通信的状态向量进入接收方时，应该落在推理过程的哪个语义位置？
```

## 范围

- 任务：多智能体数学推理中的答案前非文本通信。
- 对比方法：单智能体、普通多智能体文本讨论、裸潜状态注入、状态差分通信、题目词元锚定潜状态通信。
- 模型：当前工程优先使用 `Qwen2.5-7B-Instruct`，后续可扩展到同族模型或不同模型。
- 数据集或输入来源：当前工程优先使用 MATH 子集或 Math500 风格题目。
- 样本量：本报告先记录方案，实验结果暂空。
- 约束：发送方和接收方使用同一原始题目提示词；方法不依赖发送方生成特殊 XML、短标签或结构化通信请求。

## 核心判断

现有工作已经覆盖了很多“传什么”的选择：

- 传自然语言推理文本；
- 传嵌入序列；
- 传生成词元对应的隐状态差分；
- 传部分层的键值缓存；
- 传压缩后的潜在记忆或状态。

但这些工作留下了一个共同难点：

```text
接收方拿到一段非文本状态之后，应该在自己的推理过程中哪里使用它？
```

自然语言通信天然有位置和语义。接收方看到一句话，知道它大致在说哪个变量、哪个条件、哪个子问题。非文本状态没有这种天然坐标。一个向量如果只被加到最后一个词元、某个固定生成步或某个全局状态上，它可能只是扰动，而不是可定位的通信。

本方案的核心判断是：

```text
多智能体推理中最稳定、最普适的共享坐标系，不是发送方新写出来的消息，
而是所有智能体共同拥有的原始题目提示词。
```

因此，非文本通信可以不依赖新增文本消息，而是依赖原题中的词元位置来对齐。

## 大家做了什么

### 一、文本通信

文本通信方法让智能体把自己的中间结论、推理链、反驳或求助请求写成自然语言，交给其他智能体读取。普通多智能体辩论、协作讨论和中途求助都属于这一类。

这类方法的优点是：

- 通信内容可读；
- 接收方知道对方在说什么；
- 人可以检查通信过程；
- 很容易适配不同模型和不同任务。

这类方法的问题是：

- 文本可能冗长；
- 文本会丢失模型内部状态中的细粒度信息；
- 发送方写出的推理文本可能错误、过度自信或带偏其他智能体；
- 如果通信发生在最终答案之后，就容易变成投票或后验修正，而不是推理过程中的状态交互。

COTIE 这类工作把通信提前到推理中途，并把通信内容组织成结构化请求。它的关键不是简单“多传一点文本”，而是让一个智能体在尚未完成答案时向另一个智能体请求局部帮助，再选择性吸收对方回复。

### 二、非文本通信

非文本通信方法试图绕开自然语言，直接传模型内部或近内部的表示。已有方向包括：

- 用词表概率分布加权得到嵌入序列；
- 传发送方生成自然语言时的隐状态差分；
- 共享键值缓存；
- 选择部分层、部分头或部分上下文状态；
- 把多智能体记忆压缩成潜在表示。

这类方法的优点是：

- 通信载荷更密集；
- 理论上可以保留文本无法表达或不易表达的状态；
- 可以减少显式通信文本的长度；
- 有机会在答案生成前直接影响模型内部计算。

这类方法的问题是：

- 载荷不可读，难以判断是否有用；
- 如果注入位置不合适，容易变成普通扰动；
- 如果仍然依赖发送方自然语言 token 对齐，就没有完全摆脱文本载体；
- 如果要求发送方生成特殊格式，就会变成一个提示词协议，而不是普适机制。

### 三、我们此前的短文本锚点原型

此前的短文本锚点原型让发送方输出如下结构：

```text
<unit><anchor>Apply sine definition</anchor><work>private local reasoning</work></unit>
```

接收方只能看到短锚点，例如：

```text
[A1] Identify triangle type
[A2] Apply sine definition
[A3] Solve for DE
```

发送方详细推理放在 `<work>` 中，不给接收方看；脚本从 `<work>` 对应生成词元上抽取隐状态差分，并注入到接收方看到对应短锚点的位置。

这个原型有一个重要价值：它说明“潜状态需要语义落点”。短锚点提供了落点，所以状态不再只是被塞到最后一个词元上。

但它也有一个致命缺陷：

```text
短锚点由发送方按特殊提示词生成。
```

这意味着方法依赖发送方遵守某种输出格式。如果换一个推理提示词、换一个任务模板、换一种不暴露中间过程的设置，这个机制就可能失效。

因此，短文本锚点原型只能作为诊断原型，不能作为主方法。

## 大家没解决什么

已有工作尚未充分解决的是：

```text
不新增可读消息、不改变原始提示词时，非文本通信如何在接收方推理中找到语义落点？
```

这个问题可以拆成四个更具体的缺口。

### 一、通信载荷有内容，但缺少位置

发送方推理轨迹中的隐状态变化可能确实包含有用信息。例如它可能反映了某个变量的计算、某个约束的识别、某个候选答案的排除。

但是，如果接收方只在固定位置接收这个状态，例如提示词末尾或第 32 个生成步，那么这个状态并不一定进入了正确的局部计算。

问题不是“向量有没有信息”，而是：

```text
这个向量应该在接收方处理哪个题目条件时生效？
```

### 二、显式锚点有位置，但牺牲普适性

短文本锚点可以给状态提供位置。例如“Apply sine definition”让接收方知道这段状态对应三角函数关系。

但是，这要求发送方配合生成短标签。这样一来，方法成立的前提从“任意推理过程中的非文本通信”变成了“模型必须按指定格式组织推理”。

这会削弱普适性，也容易和结构化文本通信混在一起。

### 三、自然语言 token 对齐仍然依赖文本载体

状态差分通信可以把发送方自然语言消息中的每个 token 和对应差分绑定起来，再让接收方处理这些 token 时注入差分。

这种方式有清楚对齐，但它仍然把自然语言 token 作为通信载体。接收方仍然读到了发送方消息，只是额外得到潜状态辅助。

如果我们的目标是“不新增可读通信文本”，那么这种方式不是最终答案。

### 四、裸注入难以区分通信与扰动

如果一个同题发送方状态、一个无关题状态、一个随机同范数向量都能带来类似准确率变化，那么很难说明方法真的在“通信”。它也许只是改变了模型采样路径或扰动了内部状态。

因此，一个更强的方法必须能说明：

```text
状态载荷的作用依赖其与题目词元的语义对齐，而不是任意向量扰动。
```

## 我们提出了什么

本方案提出：

```text
题目词元锚定的潜状态通信。
```

一句话描述：

```text
不让智能体发送新的可读消息，而是把发送方推理过程中产生的隐状态变化，
根据它与原始题目词元的关联，注入到接收方读取同一题目词元的位置。
```

更具体地说：

- 发送方使用原始提示词正常推理；
- 接收方也使用原始提示词正常推理；
- 方法不改变发送方提示词；
- 方法不要求发送方输出特殊结构；
- 方法不向接收方添加新的可读通信文本；
- 通信对象是发送方推理轨迹中的潜状态变化；
- 通信坐标系是原始题目中的词元位置；
- 注入位置是接收方预填充原题时的对应词元位置。

这个方案的核心反叛点是：

```text
大家都在设计新的智能体消息；我们不发新消息。
大家都在让模型解释自己想到了什么；我们不要求模型解释。
我们只利用所有智能体已经共享的题目，把题目本身变成非文本通信接口。
```

## 方法总览

一次完整通信包含六个阶段：

| 阶段 | 输入 | 操作 | 输出 |
| --- | --- | --- | --- |
| 发送方普通推理 | 原始题目提示词 | 不改提示词，正常生成解题轨迹 | 发送方生成词元和各层隐状态 |
| 推理轨迹切段 | 发送方生成词元、隐藏状态 | 按固定窗口、句子边界或数学符号边界切成局部片段 | 若干发送方推理片段 |
| 片段到题目词元归因 | 推理片段、原题词元 | 估计该片段主要关联原题中的哪些词元 | 片段到题目词元的权重分布 |
| 潜状态载荷构造 | 每个推理片段的隐状态轨迹 | 计算隐状态变化量并裁剪范数 | 片段级潜状态载荷 |
| 接收方题目预填充注入 | 接收方原始题目提示词、锚定载荷 | 在接收方读到对应原题词元时注入载荷 | 被增强的接收方题目表示 |
| 接收方正常生成答案 | 增强后的内部状态 | 不添加可读通信文本，继续解题 | 接收方最终答案 |

## 具体实现

### 第一步：发送方正常推理

发送方输入保持原样。假设原始题目提示词为：

```text
Problem:
<题目文本>

Solve the problem. End with <answer>final answer only</answer>.
```

方法不添加如下内容：

```text
请分成 2 到 4 个步骤。
请生成锚点。
请使用 XML。
请写出局部标签。
```

运行时记录：

| 对象 | 记录内容 |
| --- | --- |
| 原始题目词元 | 词元编号、字符跨度、是否属于题目正文 |
| 发送方生成词元 | 生成顺序、文本、字符位置 |
| 发送方隐状态 | 指定层上每个生成步的 hidden state |
| 发送方注意力 | 生成词元对原始题目词元的注意力权重，若框架可取 |

如果注意力权重难以可靠获取，第一版可以只记录隐状态，再用相似度近似归因。

### 第二步：切分发送方推理轨迹

发送方生成轨迹需要被切成局部片段。每个片段代表一小段局部推理状态。

第一版可以使用固定窗口：

```text
每 16 个生成词元切一段，窗口之间不重叠。
```

更稳的版本可以使用混合规则：

| 规则 | 含义 |
| --- | --- |
| 固定长度 | 每 16 或 32 个词元形成一个片段 |
| 句子边界 | 遇到句号、换行、分号时允许提前截断 |
| 数学边界 | 遇到等号链、因此、所以、答案候选变化时形成边界 |
| 最大长度 | 单片段不超过指定词元数，避免载荷过宽 |
| 最小长度 | 片段太短则与相邻片段合并，避免噪声 |

第一版不需要完美切分。它只需要保证每个片段足够局部，不把整条推理轨迹压成一个全局向量。

### 第三步：把每个推理片段锚定到原题词元

这是本方案最关键的一步。

目标是为每个发送方推理片段得到一个分布：

```text
该片段主要对应原题中的哪些词元？
```

#### 方案 A：注意力归因

如果模型运行时可以取到注意力权重，则对片段中生成词元到原题词元的注意力做平均。

设：

- 原题词元为 `x_1, x_2, ..., x_m`；
- 发送方某个推理片段包含生成词元 `y_a, ..., y_b`；
- 第 `t` 个生成词元对第 `i` 个原题词元的注意力为 `A(t, i)`。

则片段对原题词元的关联分数为：

```text
score(i) = mean_{t=a..b} A(t, i)
```

然后选出分数最高的若干题目词元：

```text
Top-k(score)
```

为了避免锚定到提示词模板，候选词元只允许来自题目正文，不允许来自“Problem”“Solve the problem”等指令文本。

#### 方案 B：隐状态相似度归因

如果注意力取不到或成本太高，可以用隐状态相似度。

做法：

1. 对原题每个词元，在发送方预填充阶段取指定层隐状态 `h_x_i`。
2. 对推理片段取平均隐状态变化 `z_s`。
3. 计算 `z_s` 与每个题目词元表示的相似度。
4. 选相似度最高的题目词元作为锚定位置。

形式上：

```text
score(i) = cosine(z_s, h_x_i)
```

这个方案更容易实现，但解释力弱于注意力归因。

#### 方案 C：文本回退归因

如果发送方生成文本中直接出现了题目中的实体、数字、变量或公式片段，可以用字符串匹配作为回退。

例如推理片段中出现：

```text
sin 45
DE
sqrt(102)
```

则优先锚定到原题中相同或高度相似的片段。

文本回退不能作为主机制，因为它又引入了可读文本依赖；但作为调试和错误分析工具有用。

### 第四步：构造潜状态载荷

对每个推理片段，构造一个潜状态载荷。

第一版使用隐状态差分：

```text
delta_t = h_t - h_{t-1}
z_s = mean(delta_t), t 属于当前片段
```

其中：

- `h_t` 是发送方在指定层第 `t` 个生成词元处的隐状态；
- `delta_t` 表示相邻生成步的状态变化；
- `z_s` 是当前片段的平均状态变化。

为了避免注入过强，需要做范数裁剪：

```text
z_s = clip(z_s, max_norm)
```

第一版建议只使用一层，例如此前状态差分工作和本地实验都涉及的第 22 层。后续再扩展到多层。

载荷记录字段：

| 字段 | 含义 |
| --- | --- |
| `source_agent` | 发送方智能体编号 |
| `source_span` | 发送方生成片段的起止词元 |
| `layer` | 抽取载荷的层 |
| `vector_norm` | 裁剪前后范数 |
| `anchor_tokens` | 被锚定的原题词元编号 |
| `anchor_weights` | 每个原题词元对应权重 |
| `attribution_method` | 注意力归因、相似度归因或文本回退 |

### 第五步：合并多个发送方的载荷

如果有多个发送方，接收方可能在同一个原题词元位置收到多段载荷。

第一版使用保守合并：

```text
z_i = mean(z_i_from_peer_1, z_i_from_peer_2, ...)
z_i = clip(z_i, max_norm)
```

其中 `i` 是原题词元位置。

如果载荷有权重，则使用加权平均：

```text
z_i = sum_j weight_j * z_j / sum_j weight_j
```

如果不同发送方对同一位置给出方向差异很大的向量，可以记录冲突度：

```text
conflict = 1 - mean_pairwise_cosine(z_j)
```

第一版可以只记录冲突度，不使用它做门控。这样避免把方案重新变成“审计或门控”主线。

### 第六步：接收方读取同一题目时注入

接收方提示词保持原样。它不会看到任何新增通信文本。

当接收方预填充原始题目提示词时，如果当前词元位置 `i` 有来自发送方的载荷 `z_i`，则在指定层施加：

```text
h_i = h_i + alpha * z_i
```

其中：

- `h_i` 是接收方在该层处理第 `i` 个原题词元时的隐状态；
- `z_i` 是聚合后的发送方潜状态载荷；
- `alpha` 是注入强度。

为了降低破坏风险，第一版设置：

```text
alpha = 0.03
max_norm = 1.0
```

这些数值不是理论最优，只是低强度安全起点。后续需要用小样本扫描确认。

### 第七步：接收方正常生成答案

注入只发生在接收方读取题目提示词时。进入生成答案阶段后，接收方按普通方式生成。

第一版不在接收方生成过程中持续注入，原因是：

- 题目词元位置是本方法的坐标系；
- 生成阶段的词元不再是所有智能体共享的稳定坐标；
- 持续注入更容易退化成普通激活扰动。

如果后续实验显示题目预填充注入太弱，再考虑在接收方生成早期追加一次全局低强度注入，但这不属于主方法。

## 与已有工作的区别

### 与 COTIE 的区别

COTIE 的通信内容是结构化文本请求。调用方在推理中途明确写出任务片段、已有推理摘要、当前结论、求助点和期望回复格式。

本方案不让发送方写请求，也不让接收方读到发送方中间推理。接收方只看到原始题目提示词。通信内容是发送方推理轨迹中的潜状态变化。

区别可以写成：

| 项目 | COTIE | 本方案 |
| --- | --- | --- |
| 是否新增可读消息 | 是 | 否 |
| 是否改变推理协议 | 是，需要中途请求和整合 | 否，发送方和接收方提示词保持原样 |
| 通信载体 | 结构化文本 | 隐状态变化 |
| 对齐方式 | 文本字段和请求边界 | 原题词元位置 |
| 接收方读到什么 | 对方请求和回复 | 原始题目，不读对方推理文本 |

### 与状态差分通信的区别

状态差分通信把发送方自然语言消息中每个 token 的状态差分，注入到接收方处理对应自然语言 token 的位置。它的对齐坐标是发送方消息 token。

本方案的对齐坐标不是发送方消息 token，而是原始题目 token。

区别可以写成：

| 项目 | 状态差分通信 | 本方案 |
| --- | --- | --- |
| 是否需要发送方自然语言消息 | 是 | 否 |
| 差分绑定对象 | 发送方输出 token | 原始题目 token |
| 接收方是否读取发送方文本 | 是 | 否 |
| 普适性前提 | 接收方 prompt 中包含发送方消息 | 多个智能体共享同一题目提示词 |

### 与短文本锚点原型的区别

短文本锚点原型要求发送方主动生成锚点，例如：

```text
Apply sine definition
```

本方案不要求发送方生成锚点。锚点来自原题中的词元，例如：

```text
sin 45^\circ
DE
sqrt(102)
```

区别可以写成：

| 项目 | 短文本锚点原型 | 本方案 |
| --- | --- | --- |
| 锚点来源 | 发送方生成 | 原始题目已有词元 |
| 是否改变 sender prompt | 是 | 否 |
| 是否新增 receiver 可见文本 | 是 | 否 |
| 是否容易退化成文本通信 | 较容易 | 较不容易 |

### 与普通激活操控的区别

普通激活操控通常是给模型施加一个预先定义或学习出来的方向，使输出更偏向某种属性、风格、事实或行为。

本方案的载荷不是预设方向，而是来自另一个智能体在同一题目上的具体推理轨迹。它的作用位置也不是任意层或任意生成步，而是与原始题目词元对齐的位置。

区别可以写成：

```text
普通激活操控：对一个模型施加行为方向。
本方案：把一个智能体的局部推理状态传给另一个智能体，并挂到共享题目坐标上。
```

## 可能解决什么

### 一、解决非文本通信的语义落点问题

非文本载荷不再被全局注入，而是绑定到原题中的具体条件、变量、数字或关系。

例如一道题包含：

```text
sin 45^\circ
DE
\sqrt{102}
```

发送方在推理中围绕三角函数关系产生的潜状态变化，会被注入到接收方处理这些题目词元的位置，而不是被注入到提示词末尾。

这使通信更接近：

```text
在你读到这个条件时，参考另一个智能体围绕这个条件形成的内部状态。
```

而不是：

```text
给你一个向量，自己看着办。
```

### 二、解决提示词普适性问题

方法不要求发送方生成特殊格式，也不要求接收方读取新增消息。只要多个智能体共享同一题目提示词，就可以构造共同坐标系。

这比短文本锚点原型更普适。

### 三、减少与结构化文本通信的混淆

接收方没有看到发送方的中间文字推理。即使发送方生成了普通解题文本，那些文本也只用于运行时抽取状态，不作为通信文本传给接收方。

因此，如果方法有效，更能说明非文本载荷本身参与了接收方推理，而不是发送方自然语言解释帮了忙。

### 四、为后续机制分析提供更清楚的单位

由于每段潜状态都绑定到原题词元，后续可以分析：

- 哪些题目词元最常接收有用载荷；
- 数字、变量、公式、条件词哪个更适合成为注入位置；
- 正确发送方和错误发送方在锚定位置上是否不同；
- 通信成功样本是否集中在某些题型或某些语义位置。

## 实验设计

### 基础条件

第一批实验建议至少包含以下条件：

| 条件 | 含义 | 目的 |
| --- | --- | --- |
| 单智能体 | 每个智能体独立解题，不通信 | 基础能力 |
| 普通多数投票 | 多个智能体独立解题后多数投票 | 常规多智能体基线 |
| 裸状态注入 | 发送方状态聚合后注入提示词末尾 | 检查无位置对齐的潜状态是否有效 |
| 题目词元锚定注入 | 本方案 | 检查题目词元对齐是否带来额外收益 |
| 随机同范数题目词元注入 | 在相同题目位置注入随机同范数向量 | 检查收益是否只是扰动 |

| 锚定类型分布 | 载荷主要落在数字、变量、公式、条件词还是普通词 |

| 现象 | 含义 |

## 失败模式

### 一、锚定不准

如果推理片段被锚定到无关题目词元，注入就会变成位置错配。表现可能是：

- 主方法不高于裸状态注入；
- 注入位置集中在停用词或普通指令词；
- 注意力集中到提示词模板而不是题目正文。

### 二、载荷太弱

如果注入强度过低或差分均值抵消，接收方几乎不受影响。表现可能是：

- 答案与单智能体完全一致；
- 救助率和污染率都接近零；
- 主方法与无通信条件无差距。

### 三、载荷太强

如果注入强度过高，模型可能被破坏或被错误发送方带偏。表现可能是：

- 污染率明显上升；
- 输出格式异常；
- 随机同范数注入也显著改变答案。

### 四、归因方法只捕捉表面词重合

如果锚定主要依赖字符串重合，它可能只是在捕捉发送方文本里出现了哪些题目词，而没有真正捕捉内部推理关联。

这会削弱“潜状态通信”的论断。

### 五、主方法与随机同范数无差距

如果题目词元锚定注入和随机同范数题目词元注入效果相同，就不能声称通信载荷有语义信息。最多只能说明题目位置扰动会影响模型。

## 当前结论

当前报告只提出方案，不给出实验结论。

现阶段可以成立的判断是：

```text
短文本锚点原型不适合作为主方法，因为它依赖发送方按特殊提示词生成锚点。
题目词元锚定方案更符合“普适非文本通信”的目标，因为它不改变发送方或接收方提示词。
```

尚未成立的判断是：

```text
题目词元锚定潜状态通信能够稳定提升多智能体推理准确率。
题目词元锚定比裸状态注入更有效。
题目词元锚定比随机同范数注入更有效。
```

这些判断需要实验结果支持。

## 可写成论文引言的压缩表述

现有多智能体推理通信方法大多沿着两条路线发展。一类方法使用自然语言交换中间结论、反驳或求助请求，具有可读性和明确语义边界，但通信冗长且容易把错误推理显式传播给其他智能体。另一类方法尝试传递嵌入、隐状态差分或键值缓存等非文本状态，以保留更密集的内部信息，但这类状态缺少自然语言所提供的语义落点：接收方即使拿到一个潜状态向量，也未必知道它应该在自己的推理过程中对应哪个题目条件、变量或局部操作。

我们提出题目词元锚定的潜状态通信。与要求智能体生成额外通信消息的方法不同，该方法不改变发送方或接收方的原始提示词，也不向接收方暴露发送方的中间推理文本。发送方按普通方式解题，系统从其生成轨迹中抽取局部隐状态变化，并根据该局部轨迹对原始题目词元的归因，将潜状态载荷绑定到题目中的具体词元。接收方在读取同一题目提示词时，于对应词元位置注入这些载荷，然后继续正常生成答案。

该方法把共享题目本身作为多智能体非文本通信的坐标系。它试图解决的不是“能否传递内部状态”这一已被多种工作探索的问题，而是“内部状态应当在接收方推理中落在哪里”。如果实验成立，它将说明非文本通信的关键不只是载荷形式，还包括载荷与接收方推理过程之间的语义对齐。
