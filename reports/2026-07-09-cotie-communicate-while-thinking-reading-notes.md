## 一、最重要的判断

这篇论文做成的关键，不是裸传隐藏状态、键值缓存或连续潜变量。它所谓的“思维链状态通信”，主要是把推理中途的状态整理成结构化的短文本请求，再交给另一个智能体处理。

也就是说，它保留的是“答案生成前通信”这个时机，但通信内容不是模型内部张量，而是一个边界清楚、可被另一个智能体直接理解的求助请求。

一条请求大致包含：

- 当前任务中与求助有关的片段；
- 到目前为止的简短推理摘要；
- 当前中间结论或部分答案；
- 此刻具体需要什么帮助；
- 希望对方提供哪类帮助；
- 这是局部咨询还是子任务注入；
- 期望回复的格式。

这和我们之前尝试的裸隐藏状态或键值缓存注入不是同一类方法。我们之前的问题是让接收方自己解释一段连续状态；这篇论文的问题是让发送方把当前推理卡点变成一个明确的问题，再让接收方给出局部帮助。

## 二、它到底怎么通信

论文把一次通信定义为发生在调用方推理尚未结束时的跨智能体请求。调用方不是等自己写完整个答案后再交流，而是在推理中途向另一个智能体请求帮助。

一次请求包含以下信息：

| 字段 | 含义 |
| --- | --- |
| 当前步数 | 调用方现在处于第几步推理 |
| 任务片段 | 与本次求助直接相关的题目或子任务内容 |
| 简短推理摘要 | 调用方已经想到了什么，但只保留必要部分 |
| 当前中间结论 | 调用方暂时得到的结论或部分答案 |
| 具体求助点 | 需要验证、计算、检索、澄清、调试或批评的具体问题 |
| 请求类型 | 局部咨询或子任务注入 |
| 回复格式 | 希望对方按什么格式回答 |

论文判断一次交流是否属于“思维链状态通信”，主要看三点：

1. 调用方还没有最终定稿答案；
2. 发出的请求来自调用方当前的中间推理状态；
3. 对方回复会在最终答案前更新调用方后续推理。

如果对方回复只用于最后投票、最后辩论或最后答案选择，论文明确认为那仍然是“答案后通信”，不属于它要研究的中途状态通信。

## 三、两类请求

论文中有两类请求。

第一类是“咨询请求”。它用于局部不确定点，例如：

- 这个代数变换是否正确；
- 这个边界情况是否会让代码失败；
- 这条证据是否支持当前答案；
- 当前解释有没有逻辑漏洞。

第二类是“注入请求”。它用于边界清楚的可分离子任务，例如：

- 让对方做一个文献综述片段；
- 让对方分析一段报错；
- 让对方写一个候选实现；
- 让对方给出数据库查询或证据综合结果。

两类请求的共同点是：对方返回的内容都不是最终答案。调用方需要先评估它，再决定哪些部分进入自己的后续推理。

## 四、它为什么能做成

### 1. 它不是多传一点，而是早一点传到正确位置

论文设置了一个很关键的对照：晚期思维链消息传递。这个对照也会传推理摘要，但它是在每个智能体完成本地推理之后才传。

结果显示，中途状态通信仍然更好：

| 任务 | 晚期思维链消息传递 | 中途状态通信 |
| --- | ---: | ---: |
| 数学 | 91.8 | 96.0 |
| 代码生成 | 94.3 | 95.6 |
| 多跳问答 | 60.3 | 65.4 |
| 多智能体编码场景 | 56.7 | 63.5 |
| 多智能体研究场景 | 87.1 | 89.3 |
| 多智能体数据库场景 | 78.6 | 82.0 |

这说明论文想证明的不是“给别人看更多推理文本就有用”，而是“对方信息必须在调用方继续推理前进入状态”。

### 2. 它有明确的选择性吸收机制

每次收到对方回复后，调用方都要记录一条整合记录。整合记录需要说明：

- 回复是完全吸收、部分吸收、拒绝，还是暂缓；
- 哪些点被接受；
- 哪些点被拒绝；
- 有没有冲突；
- 调用方当前状态被怎样更新；
- 为什么这样处理。

这非常关键。它不是简单把对方回复拼进上下文，而是要求调用方先判断、筛选、再更新状态。这个机制可以降低错误信息直接污染后续推理的风险。

### 3. 它给智能体分了角色

不同任务里，智能体有明确分工：

- 数学任务中，一个智能体负责解题，另一个负责验证；
- 代码任务中，一个负责写代码，另一个负责审查；
- 多跳问答中，一个负责检索证据，另一个负责综合答案；
- 多智能体基准中，保留原基准给定的角色和工具环境。

每个智能体都被告知：

- 如果不确定具体事实、计算、解释或实现细节，就发起咨询；
- 如果遇到边界清楚的子任务，就交给另一个智能体处理；
- 收到回复后必须先评估，再接受、拒绝或继续请求；
- 只有完全有把握时才输出最终答案。

所以它不是随机交流，而是有任务类型、有角色类型、有求助类型的中途通信。

### 4. 它没有声称发明新的摘要方法

论文特别说明，它没有提出新的摘要或压缩算法。在多智能体基准实验里，历史摘要使用的是基准自带机制。

它的核心主张只有一个：把通信附着点从“完成答案之后”移动到“推理过程之中”。

## 五、答案和轨迹长什么样

附录中的例子来自一个研究类任务，目标是围绕一个具身机器人框架形成研究提案。

### 第一次交流

调用方是第一个智能体，对方是第三个智能体，发生在第 4 步。请求类型是子任务注入。

调用方请求对方做文献综述，主题包括：

- 机器人认知架构；
- 混合控制系统；
- 机器人中的可解释人工智能；
- 人机协作中的信任；
- 大语言模型和视觉语言模型如何接入机器人架构；
- 这个机器人框架与相关工作的差异和空白。

对方回复的主要内容是：

- 已有工作包括认知架构、规范性架构、协作系统、混合控制、可解释机器人等；
- 该框架的战略层和战术层相对独立又能互动，扩展了传统混合控制；
- 当前基于大语言模型或视觉语言模型的机器人系统仍有可解释性、信任、能力校准和人机信息交换问题。

调用方的整合记录是：

- 状态：完全吸收；
- 接受：认知架构脉络、双层架构贡献、可解释性限制、能力校准、人机协作节奏不匹配、信息交换需求；
- 拒绝：无；
- 冲突：无；
- 更新后的状态：调用方把这些研究空白作为后续构思研究方向的中间状态。

### 第二次交流

调用方接着向第二个智能体请求研究想法，而且显式基于第一次交流得到的文献综述。

调用方请求的大意是：

```text
基于这个机器人框架的引言和第三个智能体做出的文献综述，
请提出能够回应这些研究空白的潜在研究想法。
```

对方提出了三个方向：

1. 面向该机器人框架的对抗鲁棒可解释规划；
2. 面向跨领域技能迁移的因果表征学习；
3. 面向人机通信的对抗感知协议。

调用方的整合记录是：

- 状态：部分吸收；
- 接受：第一个方向和第二个方向；
- 拒绝：第三个方向，因为回复被截断，不完整；
- 冲突：无；
- 更新后的状态：调用方现在有两个完整研究方向，第三个方向需要后续补全后才能使用。

### 最终答案

最终答案不是直接复制对方回复，而是把被接受的第一个方向发展成研究提案：

```text
如何为该机器人框架开发一个鲁棒的战略层规划模块，
使其在对抗条件和分布偏移下仍能保持因果可解释性，
同时确保安全的人机协作？
```

论文还进一步补了四个技术组件：

1. 结合大语言模型推理和形式化符号规划的混合规划架构；
2. 带有解释保持正则化的对抗训练；
3. 面向鲁棒性的解释生成器；
4. 将新情境映射到已知因果模式的领域适应模块。

所以它的完整轨迹可以概括为：

```text
中途请求文献综述
-> 对方返回研究空白
-> 调用方吸收空白
-> 基于这些空白请求研究想法
-> 对方返回多个方向
-> 调用方接受两个、拒绝一个
-> 最后发展其中一个被接受方向形成最终研究问题和方案
```

## 六、对我们现有尝试的启示

第一，如果我们想复现这类方法，应该先做一个结构化文本中途状态通信基线，而不是直接继续做裸隐藏状态或键值缓存注入。

第二，我们之前失败的连续状态通信，可能缺的不是“更早通信”，而是三个支架：请求边界、角色分工、整合门控。

第三，如果我们还想做潜变量或键值缓存通信，更合理的问题是：能否把这种结构化中途状态压缩进非文本通道，同时保留“接受、拒绝、冲突、状态更新”这套整合机制。

第四，这篇论文没有解决“非文本内部状态如何被可靠解释”的问题。它实际上绕开了这个难题，用结构化文本中间状态先把答案前通信做稳。
