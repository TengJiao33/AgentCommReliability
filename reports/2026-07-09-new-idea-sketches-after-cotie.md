# 撞车之后的新想法草案

日期：2026-07-09

工程状态（2026-07-20）：本文仍是设计草案，没有“可拒绝潜在通信”或完整压力测试的专用 runner/test。主想法后来收敛为 `2026-07-10-latent-communication-new-insights-and-mechanisms.md` 中的接收方条件化增量通信；两份文档属于同一研究线的前后阶段，不重复计为两个已实现方法。后续新实验默认面向 `SJTU_HPC`；登录和调度已经核验，正式 GPU 作业环境仍需确认。

## 先给结论

我现在最看好的方向不是继续声称“答案前通信”或“非文本通信”本身新，而是把问题改成：

```text
如何让非文本通信变得可拒绝、可审计、可校准？
```

更具体一点：

```text
现有非文本通信论文多关注能不能传、怎么传、传得省不省；
COTIE 证明了中途通信有用，但它靠的是结构化文本请求和显式整合记录；
MAD-M2 证明了错误记忆会伤害多智能体推理，需要过滤；
那么一个自然空白是：
非文本状态能否像 COTIE 文本回复一样，被接收方选择性接受、部分接受、拒绝或推迟？
```

这条线比“提前通信”安全，也比“传键值缓存”安全。它不是抢介质，也不是抢时机，而是抢**通信可靠性**。

## 主想法：可拒绝的潜在通信

### 一句话

让接收方不是盲目使用发送方的隐藏状态或键值缓存，而是先判断这段非文本状态是否值得信任，再决定接受、部分接受、拒绝或降权。

### 为什么它可能新

现有几条线各自缺一块：

- COTIE 有“接受、拒绝、状态更新”的整合机制，但通信内容主要是结构化文本。
- KVComm、Dense Latent Communication、LatentMAS 等有非文本通道，但通常默认传来的状态有用，重点在对齐、压缩、选择层、效率。
- MAD-M2 研究错误文本记忆会误导智能体，但它处理的是文本记忆，不是隐藏状态或键值缓存。

所以我们的切口可以是：

```text
把 COTIE 的整合记录和 MAD-M2 的记忆过滤，迁移到隐藏状态/键值缓存通信中。
```

### 方法雏形

每次非文本通信不直接注入，而是经过一个“整合门”：

1. 发送方产生某种非文本状态：隐藏状态、状态差分、键值缓存或压缩键值缓存。
2. 接收方先用自己的当前状态做一次轻量诊断。
3. 诊断结果决定这段非文本状态的命运：
   - 完全接受；
   - 只接受部分层、部分头、部分时间步；
   - 降权使用；
   - 完全拒绝；
   - 延迟到下一步再使用。
4. 接收方记录一条“非文本整合记录”：
   - 接受了哪些层或头；
   - 拒绝了哪些层或头；
   - 触发拒绝的信号是什么；
   - 接收后答案是否改变；
   - 原本正确样本是否被破坏。

### 关键实验

构造四类发送方状态：

| 状态来源 | 目的 |
| --- | --- |
| 同题正确发送方 | 测试能否救助接收方 |
| 同题错误发送方 | 测试是否会误导接收方 |
| 无关题发送方 | 测试是否能识别不相关状态 |
| 随机同范数状态 | 测试收益是否只是扰动效应 |

评价指标不要只看准确率，而要看：

| 指标 | 含义 |
| --- | --- |
| 救助率 | 原本错误的接收方被通信改对的比例 |
| 污染率 | 原本正确的接收方被通信改错的比例 |
| 净收益 | 救助率减去污染率 |
| 过度信任率 | 接收方接受了错误或无关状态的比例 |
| 拒绝准确率 | 整合门是否能拒绝有害状态 |
| 状态贡献率 | 接收方答案改变中有多少能归因于通信 |

### 论文题目候选

```text
When Should Latent Messages Be Rejected?
Auditing and Gating Non-Text Communication in LLM Multi-Agent Reasoning
```

中文说法：

```text
非文本通信不只是要能传，还要能拒绝。
```

## 第二个想法：潜在通信压力测试

### 一句话

不要先发明新方法，先做一个系统诊断：现有隐藏状态和键值缓存通信在什么条件下帮助、伤害、失效。

### 为什么有价值

Dense Latent Communication 提醒了一个关键点：接收方能不能看到题目，会改变通道需要承载的信息量。接收方看得到题目时，非文本通道可能只是在提供稀疏推理信号；接收方看不到题目时，通道必须携带完整任务信息。

我们可以把这个思想扩展成一套压力测试：

| 设置 | 问题 |
| --- | --- |
| 接收方看得到题目 | 通信是否只是轻微 steering |
| 接收方看不到题目 | 通信是否真能承载任务信息 |
| 发送方正确 | 通道是否能传递有用信号 |
| 发送方错误 | 通道是否会造成错误锚定 |
| 发送方与接收方模型不同 | 通道是否依赖同构空间 |
| 通信发生在早期、中期、晚期 | 哪个时机最容易污染或救助 |

这可以写成 finding paper：

```text
Rethinking Latent Communication:
When Hidden States and KV Caches Help, Hurt, or Merely Steer
```

这条线的好处是，即使方法没有显著提升，也能产出有价值结论。

## 第三个想法：文本契约，潜在载荷

### 一句话

让文本只负责描述“我要解决什么局部问题”，让非文本状态负责携带“求解所需的高维证据”。

### 动机

COTIE 成功，是因为请求边界清楚；非文本通信高效，是因为载荷密集。那我们可以把二者拆开：

- 文本契约：说明任务片段、当前不确定点、期望帮助类型；
- 潜在载荷：隐藏状态或键值缓存，作为补充证据；
- 整合门：检查潜在载荷是否真的支持文本契约。

这不是纯文本 COTIE，也不是裸键值缓存通信，而是：

```text
结构化文本请求 + 非文本证据载荷 + 可拒绝整合
```

### 可能的实验

对比四种方法：

| 方法 | 通信内容 |
| --- | --- |
| 纯文本中途请求 | 类似 COTIE |
| 裸键值缓存 | 没有文本契约 |
| 文本请求加随机载荷 | 控制潜在载荷是否真有信息 |
| 文本请求加真实潜在载荷 | 我们的方法 |

如果真实潜在载荷在相同文本请求下仍有额外收益，就说明非文本通道不仅是噱头。

## 第四个想法：把“什么时候通信”改成“什么时候不通信”

### 一句话

不要再问什么时候应该通信，而是问：什么时候通信会伤害推理？

### 为什么更锋利

“什么时候通信”已经被很多工作碰过：通信调度、门控、注意力选择、层选择、COTIE 中途请求。

但“什么时候应该拒绝通信”更少被正面研究，尤其是非文本通信。

可研究的触发信号：

- 接收方对自己答案很有把握时，不接受外部状态；
- 发送方状态与接收方当前方向冲突太大时，只部分接受；
- 发送方状态来自高困惑度轨迹时拒绝；
- 发送方和接收方对关键中间变量不一致时，先请求文本澄清；
- 多个发送方状态相互冲突时，选择最保守整合。

这个方向的标题可以更狠：

```text
The Right to Ignore:
Refusal Mechanisms for Latent Communication in LLM Agents
```

## 第五个想法：潜在记忆免疫系统

### 一句话

把错误潜在状态当成“污染记忆”，训练或设计一个免疫系统去识别它。

### 灵感来源

MAD-M2 说错误文本记忆会误导多智能体推理。非文本状态更危险，因为它不容易被人或模型明确检查。

我们可以构造“潜在污染源”：

- 错题键值缓存；
- 同题但错误解题轨迹；
- 正确题但错误中间步骤；
- 截断的状态；
- 层错位状态；
- 位置错位状态；
- 高噪声状态；
- 对抗性状态。

然后让接收方学习或启发式判断：

```text
这段状态像不像会污染我？
```

这个方向比较天马行空，但很有论文味，因为它把“通信”转成了“安全性和鲁棒性”问题。

## 第六个想法：潜在通信后的自然语言审计

### 一句话

接收方使用非文本状态后，必须生成一条简短审计说明：这段状态改变了我什么。

### 为什么有用

非文本通信最大的问题是不可解释。COTIE 的整合记录很好，因为人能看到接受了什么、拒绝了什么。非文本通道也许可以借鉴这个机制：

```text
我接受了发送方状态，因为它支持了变量 x 的计算；
我拒绝了发送方状态，因为它与我的约束 y 冲突；
我只接受了中间层状态，因为后层状态过度指向某个答案。
```

即使这条审计说明不是完美解释，也可以作为诊断工具。我们可以比较：

- 有审计说明时，污染率是否降低；
- 审计说明是否能预测通信是否有害；
- 审计说明和真实答案变化是否一致。

这条线适合和可拒绝潜在通信合并。

## 我最推荐的组合

不要一次做六个方向。最稳的组合是：

```text
主问题：
非文本通信什么时候应该被拒绝？

方法：
给隐藏状态/键值缓存通信加一个选择性整合门。

诊断：
用正确、错误、无关、随机四类发送方状态做压力测试。

对照：
裸非文本通信、COTIE 式文本中途通信、MAD-M2 式过滤思想。

贡献：
提出非文本通信的救助率、污染率、过度信任率、拒绝准确率。
```

这可以包装成一篇比较稳的 finding paper，也有机会扩成方法论文。

## 最小可行实验

先不要上复杂训练。先做一个小矩阵：

| 条件 | 说明 |
| --- | --- |
| 无通信 | 基线 |
| 裸状态注入 | 复用我们已有失败/弱结果 |
| 错误状态注入 | 看污染率 |
| 状态差分注入 | 对齐 SDE |
| 键值缓存注入 | 对齐 KVComm |
| 简单门控注入 | 只在接收方低置信度时接受 |
| 冲突门控注入 | 发送方和接收方中间答案冲突时降权或拒绝 |

第一阶段只需要回答：

```text
门控是否能降低污染率，而不显著牺牲救助率？
```

如果答案是肯定的，这个 idea 就有生命。
