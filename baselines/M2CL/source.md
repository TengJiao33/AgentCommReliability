## Why Pick This Up

M2CL is a useful pressure object after the communication-regimes synthesis because it moves the center of gravity from message retention to context alignment. Instead of only asking which peer messages are visible, it asks what role/context state each agent should inhabit at each discussion round.

This makes it directly relevant to the current concern that recent DAR variants are too shallow: M2CL points at a deeper variable that our traces do not yet record.

## Known Caveats

- `method/M2CL.py::gen_response` immediately returns the fixed string `"success generate"`, making the later local Llama/OpenAI generation branches unreachable.
- `main.py` requires `api_key.txt` even for local-model paths.
- `main.py` hardcodes `http_proxy` and `https_proxy` to `http://127.0.0.1:21882`.
- `main.py` expects `dataset/<dataset>/question_answer.npy`, but no dataset folder is present in the inspected checkout.
- `README.md` still lists model checkpoint release as TODO.
- `method/M2CL.py` uses list multiplication for buffers such as `context_buffer = [[]] * len(self.agent_list)`, which aliases inner lists.
- Full reproduction likely requires substantial patching before it would test the intended method.

## Loose Threads

- Treat M2CL first as a source of trace fields: generated context instruction, role/context diversity, and context coherence.
- Revisit runnable reproduction only after checkpoint/data availability is clear or after a deliberately scoped synthetic smoke is designed.
- If patching, separate mechanical run fixes from method-changing fixes such as removing the `gen_response` stub.
