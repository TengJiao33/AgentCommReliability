# HSA 支撑型窄补全组件说明

日期：2026-06-19

## 核心判断

支撑型窄补全目前是 HSA-v0 中最值得推进的方法组件候选。它把三十六行 HSA 真跑的硬准入链路从 `35/36` 推到 `36/36`，多余最终卡片从 `40` 到 `42`，仍远低于全范围信息控制的 `195`。它在旧 33 行和 15 行包上重放干净，暂时没有发现过度补卡。

它当前仍属于诊断后处理。若要进入论文主方法，需要把输入字段、补全条件、禁区和失败标准固定下来，并在更大 HSA 派生包上压力测试。

## 组件要解决的问题

HSA 的长档案任务暴露了一个召回缺口：模型能选出最终答案，也能提出部分阻断证据，却漏掉支持该答案的多张可见、已验证、面向最终决策者的支撑卡。`baker_2010` 基础行就是这个情况：模型选择 Roberts，但少提了 Roberts 的协作、教学、研究产出、多样性和基金支撑卡，导致硬准入因证据不足退回。

这个缺口和扰动行的错误承诺不同。扰动行主要靠硬准入拦截未验证、越界或不充分证据；长档案基础行需要在模型已经作出具体选择时，补齐同一选择下的可审计支撑卡。

## 允许输入

补全器只能读取预测时已经可见或由模型自己输出的字段：

| 来源 | 字段 |
| --- | --- |
| 输入包 source cards | `card_id`、`source_id`、`source_role`、`recipient_scope`、`verification_state`、`evidence_type`、`content` |
| 模型输出 | `final_decision`、`candidate_units`、每个候选单元的 `recipient` 和 `card_ids` |
| 输入包任务字段 | `final_decider` |

补全器不能读取评价专用字段，包括 `required_slots`、`acceptable_card_ids`、`expected_final_decision`、`gold_answer`、`oracle_unit_ids` 和 `downstream_scoring_obligations`。

## 规则

第一，解析模型输出。若模型输出不能解析成对象，补全器不新增任何单元，只记录解析错误。

第二，确定最终决策者。默认使用输入包中的 `final_decider`，缺失时才回退到 `final_decider` 字符串。

第三，保留已有候选证据。已经被模型分配给最终决策者的卡片不会重复新增。

第四，阻断卡补全始终启用。满足以下条件的卡片会被补到最终决策者候选单元中：

| 条件 | 要求 |
| --- | --- |
| 证据类型 | `blocker` |
| 验证状态 | `verified` |
| 可见范围 | `final_decider` 在 `recipient_scope` 中 |
| 去重 | 该卡片尚未出现在最终决策者候选单元中 |

第五，支撑卡补全只在支撑型策略中启用。满足以下条件的卡片会被补到最终决策者候选单元中：

| 条件 | 要求 |
| --- | --- |
| 证据类型 | `support` |
| 验证状态 | `verified` |
| 可见范围 | `final_decider` 在 `recipient_scope` 中 |
| 模型选择 | `final_decision` 是具体答案，且不是证据不足 |
| 匹配方式 | 标准化后的模型答案能在 `card_id`、`content` 或 `source_id` 中匹配 |
| 去重 | 该卡片尚未出现在最终决策者候选单元中 |

第六，新增单元只携带可审计说明。阻断卡优先级为 `9.5`，支撑卡优先级为 `9.0`；新增单元的说明只声明“保留可见、已验证、范围内且匹配模型选择的卡片”。

## 禁区

补全器不得新增未验证卡片。

补全器不得新增最终决策者不可见的卡片。

补全器不得在模型输出证据不足时新增支撑卡。

补全器不得根据标准答案、必需槽位或评测器可接受卡片反推要补哪些卡。

补全器不得修改模型的最终选择。

补全器不得把其他候选人的支撑卡补给当前模型选择。

## 当前证据

| 包 | 原始硬准入 | 阻断补全 | 支撑型窄补全 | 多余最终卡片 | 读法 |
| --- | ---: | ---: | ---: | ---: | --- |
| 15 行 | `14/15` | `15/15` | `15/15` | `10` | 无新增过度补卡 |
| 33 行 | `32/33` | `33/33` | `33/33` | `37` | 无新增过度补卡 |
| 36 行 | `34/36` | `35/36` | `36/36` | `42` | 修掉 Roberts 支撑卡缺口 |

三十六行包的全范围信息控制多余最终卡片是 `195`。支撑型窄补全后的 `42` 说明当前规则没有靠全收档案卡拿分。

## 论文中的身份

当前最稳妥的写法是把它列为“可审计补全组件候选”。主表应同时报告四条链路：模型直出、硬准入、阻断补全、支撑型窄补全。这样读者能看到收益来自哪个环节。

若更大包继续满足以下条件，它可以升为正式方法组件：

| 条件 | 升级信号 |
| --- | --- |
| 扰动行 | 仍保持证据不足，不因支撑补全产生错误承诺 |
| 基础行 | 修复长档案支撑召回缺口 |
| 多余卡片 | 明显低于全范围信息控制 |
| 旧包重放 | 不降低 15 行和 33 行成绩 |
| 字段审计 | 静态检查确认没有读取评价专用字段 |

若出现以下情况，它只能保留为诊断工具：

| 情况 | 处理 |
| --- | --- |
| 在扰动行补出错误答案 | 暂停进入主方法 |
| 多余最终卡片接近全范围信息控制 | 作为过度准入失败记录 |
| 依赖任务特定名字匹配 | 改成更严格的候选实体规范后再测 |
| 更大包无法复现收益 | 保留为 `baker_2010` 个案修复 |

## 下一步压力测试

下一步先固定这个规则说明和主表身份，再扩 HSA 派生包。扩包时必须保留同一组对照：共享信息、全范围信息、理想准入、模型直出、硬准入、阻断补全和支撑型窄补全。核心读数是严格正确、基础行正确、扰动行正确、槽位召回、多余最终卡片和强制作答。
